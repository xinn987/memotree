# 产品设计

## 第一批用户

MVP 面向爸妈、爷爷奶奶，其中老人是核心用户。产品不应表现为网盘、NAS、全量备份工具或素材管理器，而应表现为打开就能看宝宝近况的家庭精选照片流。

MemoTree 的主心智是“家人每天打开看看宝宝近况”。全量照片备份由其他工具或人工流程承担，MemoTree 只承载家庭成员愿意共享给家人看的精选照片和视频。

## 核心优先级

1. 首屏看图：打开首页直接进入家庭精选近况时间线。
2. 上传可见但次要：保留明显上传入口，但不抢占浏览内容。
3. 私密访问：非家庭成员不可访问，移除成员后立即失去访问能力。
4. 原文件兜底下载：下载不是核心心智，但后端必须从第一版支持权限校验和短期授权。

## 家庭空间

MVP 使用 `Family` 作为私密共享空间和权限边界，不引入宝宝、相册或文件夹层级。

- `Family` 表示一个家庭精选近况空间，不表示某一个宝宝。
- `User` 和 `Family` 是独立领域对象，通过 `FamilyMember` 形成多对多关系。
- `FamilyMember` 表示用户在某个家庭里的称呼、角色和状态。
- MVP 不引入 `Child` 模型，也不要求上传时选择具体宝宝。
- MVP UI 默认按单家庭体验设计，但领域模型允许一个用户加入多个家庭。

## 时间线

首页主时间线表达宝宝和家庭近况的成长顺序，优先按拍摄时间展示。

- 首页按 `captured_at` 倒序排列，并按月份、日期分组。
- 如果媒体缺失拍摄时间，则使用 `uploaded_at` 混入时间线。
- 上传时间排序后续可作为“最近添加”视图补充，不进入 MVP 主流程。
- 详情页可以展示拍摄时间、上传者和上传时间。

## 媒体资产

上传到 MemoTree 的内容属于家庭空间，上传者只是贡献者和审计信息。

- `MediaAsset` 表示时间线上的一个媒体条目，可以是 `photo`、`video` 或 `live_photo`。
- 一次批量上传会创建多个独立 `MediaAsset`，不引入帖子、相册、批次或动态分组。
- 时间线永远展示 `MediaAsset`，不会直接展示原始文件。
- `MediaOriginal` 保存用户上传的原文件，例如 HEIC、JPG、MOV 或 MP4。
- `MediaRendition` 保存给前端展示用的派生资源，例如 `thumbnail`、`display_image` 和 `display_video`。
- 原文件用于私有存储和授权下载，展示资源用于时间线和详情页渲染。
- iPhone 实况照片合并为一个 `live_photo`，包含图片原文件和视频原文件；MVP 默认展示静态图片，不把实况视频单独展示到时间线。
- `admin` 可以软删除家庭媒体；普通 `member` 不能删除媒体。
- 软删除后的媒体不出现在时间线、详情和下载接口中，但原文件可等待后台或人工清理。

## 成员与权限

MVP 只区分管理员和普通成员。

- `admin` 可以邀请成员、撤销未使用邀请、移除成员、管理家庭基础信息和成员称呼。
- `member` 可以查看时间线、查看媒体详情、上传照片/视频和下载原文件。
- 媒体访问只要求当前用户是 active member。
- 管理动作要求当前用户是 active admin。
- 移除成员只移除其家庭成员身份，不删除全局用户，也不删除其历史上传内容。
- 应避免家庭空间失去最后一个 active admin。

## 登录与邀请

MVP 采用持久化账号 + 密码登录，并通过家庭邀请控制首次加入。邀请负责判断用户是否有资格加入某个家庭，账号密码负责后续重复登录，浏览器会话只负责减少重复输入。

- 用户账号长期落库。
- 用户通过登录名和密码证明自己是同一个全局账号。
- MVP 的登录名可以优先使用手机号，但手机号在未接入短信验证码前只作为登录名，不作为强身份验证。
- 管理员创建一次性邀请。
- 一个邀请链接最多成功绑定一个账号。
- 用户通过邀请获得家庭成员权限。
- 被移除成员不能使用旧邀请重新加入，必须重新邀请。
- 用户忘记密码时，MVP 可以先由家庭管理员重置密码或重新邀请，短信/邮箱找回后置。

## 暂不做

- 评论、点赞、收藏。
- AI 整理、人脸识别、成长报告。
- 文件夹、标签、复杂搜索。
- 自动同步手机相册。
- 复杂角色权限。
- 宝宝模型和按宝宝筛选。
- 帖子、动态分组、上传批次作为浏览对象。
- 实况照片动态播放。
