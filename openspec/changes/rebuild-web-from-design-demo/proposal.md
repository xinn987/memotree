## Why

当前 `web/` 前端虽然已经覆盖 MemoTree MVP 的主要业务流程，但仍是单文件、后台面板式实现，视觉、信息层级和交互节奏都没有落地已经确认的 `design/demo`。现在需要以 demo 为唯一设计基线完整重写前端，让真实产品呈现为温暖、私密、照片优先的家庭相册，同时建立可持续维护的正式工程结构。

## What Changes

- **BREAKING**：移除对现有前端组件结构、className、页面布局和内部状态组织的兼容要求，重写 `web/src/` 的应用结构与表现层。
- 按 `design/tokens.css`、`design/demo/assets/demo.css` 和七个 demo 页面重建登录/注册、邀请加入、时间线、上传、邀请管理、成员管理和媒体详情。
- 建立正式的前端路由、会话守卫、API 客户端、功能模块、共享组件、样式分层和纯函数工具组织。
- 严格复刻 demo 的布局、字体层级、配色、间距、圆角、阴影、响应式行为、动效和中文文案语气。
- 保留现有后端 API 契约和已实现业务行为，不因视觉重构轻易修改服务端。
- demo 已展示但后端尚未提供的能力保留页面位置和完整视觉状态，以明确的占位交互呈现，不伪造成功结果，也不调用不存在的接口。
- 补齐加载、空状态、错误、禁用、键盘焦点和减少动效等正式产品状态。

## Capabilities

### New Capabilities

- `design-demo-web-experience`: 定义 MemoTree 生产前端按 `design/demo` 完整重写后的页面结构、视觉保真、路由、响应式行为、占位能力和工程质量要求。

### Modified Capabilities

- None.

## Impact

- 主要影响 `web/src/`、`web/package.json` 及前端构建配置；旧前端模块可以删除或整体替换。
- `design/` 继续作为视觉真源，原则上不随实现改写。
- 现有 Go API、Worker、数据库、对象存储和部署接口不做行为性修改。
- 前端将新增正式路由依赖，并可按验证需要补充轻量测试依赖；不引入与目标无关的重型框架。
- 对后端缺失能力的任何真实接入都需要另行确认或建立后续 change。
